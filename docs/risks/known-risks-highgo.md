# 瀚高 v4.1.5 已知风险库

> 按严重度分级，Pilot 中发现的新风险回灌到本文件。
> 严重度：🔴 高（阻塞 / 数据错误） / 🟡 中（需改造 / 影响功能） / 🟢 低（注意即可）

## R-001 🔴 JDBC 驱动坐标与版本（⚠️ C2 待核实）

**风险**：瀚高 JDBC 驱动（HighGo JDBC）通常不在 Maven 中央仓库，GroupId / ArtifactId / Version 坐标及官方下载路径暂未核实（C2 待核实），依赖公司内部 Nexus / Artifactory 托管，或需人工 `mvn install:install-file`。

**影响**：Stage 2 依赖替换阻塞；新成员本地环境无法复现；CI 拉取失败。

**缓解**：
- Stage 0 确认公司内部仓库是否已托管瀚高驱动
- 文档化官方驱动下载链接与本地 `mvn install:install-file` 命令（含校验 sha256）
- 驱动版本写死（禁用 `LATEST` / `RELEASE`），在 `dependencyManagement` 锁定
- 核实完成后回灌到 `docs/references/` 并关闭 C2

## R-002 🔴 厂家兼容脚本 DATE_FORMAT 递归风险（⚠️ C10 待核实）

**风险**：瀚高官方/厂家提供的 MySQL 兼容脚本中，`DATE_FORMAT` 等函数的实现可能内部递归调用自身名称（同名函数/宏），在特定参数或嵌套调用下可能导致无限递归 / 栈溢出。C10 暂未核实具体脚本版本的实现细节。

**影响**：运行时堆栈溢出、连接挂死、慢 SQL 雪崩。

**缓解**：
- Stage 0 审阅兼容脚本源码，重点看 `DATE_FORMAT`、`STR_TO_DATE`、`FROM_UNIXTIME` 等涉及字符串格式化的函数
- Pilot 中对高频 `DATE_FORMAT` 路径做压测与递归深度观测
- 若确认递归，fix-issue 记录并改写为 `to_char` 原生调用
- C10 核实后回灌到 `docs/references/mysql-to-highgo-function-mapping.md`

## R-003 🟡 Druid SQL Parser 对瀚高支持有限

**风险**：Druid `wall filter` / SQL 解析对瀚高（PG 系）非标语法支持不完整，可能误报或解析失败，监控页异常。

**影响**：启动 WARN、`wall` 误拦合法 SQL、监控失效。

**缓解**：
- `filters` 初期移除 `wall`，仅保留 `stat`
- `db-type` 显式设为 `postgresql`
- 观察启动日志与监控页，逐步开启必要 filter
- 严重兼容问题上升为阻塞项

## R-004 🟡 时区与 TIMESTAMP 语义

**风险**：MySQL `TIMESTAMP` 会自动做 UTC 存储、读取时按 session 时区转换；瀚高的 `timestamp` 与 `timestamptz` 语义不同——不带时区的字段不做转换，带时区的走 PG 语义。应用若强依赖 MySQL "存 UTC 取本地"的隐式行为会踩坑。

**影响**：时间字段显示偏差、跨时区比较错误、定时任务触发时间漂移。

**缓解**：
- 库级会话时区显式设置为 `Asia/Shanghai`
- 盘点所有 `TIMESTAMP` 字段，确认使用 `timestamp` 还是 `timestamptz`
- Pilot 写专项测试覆盖时间写入/读取路径
- 应用层时间优先用 `LocalDateTime` + 显式时区换算

## R-005 🟡 大小写敏感与标识符

**风险**：瀚高（PG 系）默认将未加引号的标识符转小写，若历史 Schema / SQL 存在混用大小写或带引号的驼峰命名，会出现 `relation not found`。

**影响**：`Table not found` / `column does not exist` 类错误。

**缓解**：
- 统一全小写 snake_case 标识符
- Schema 导出后全量转小写
- Mapper XML / 代码 SQL / 动态 SQL 全量同步
- 禁止在 DDL 中使用双引号驼峰命名

## R-006 🟡 PG 保留字与 MySQL 差异

**风险**：瀚高属 PG 系，`user` / `type` / `role` / `order` 等在 PG 中为保留字或强关键字，MySQL 中作为列名/表名可用的在瀚高下需加引号或改名。

**影响**：DDL 或 DML 失败、动态 SQL 拼接失效。

**缓解**：
- 统一策略：冲突列名改名（推荐）或双引号包裹
- Pilot 中先列出本工程全部保留字冲突清单
- Mapper XML 中改用 `user_name` 等明确命名

## R-007 🟡 连接池资源配置

**风险**：瀚高连接数配额与 MySQL 差异较大，同参数下易出现连接耗尽或资源浪费。

**影响**：高并发连接失败、`too many clients` 错误。

**缓解**：
- 按瀚高实际配额重新测算 `maximum-pool-size`，通常下调 30%~50%
- 监控 `pg_stat_activity` 观察真实连接占用
- 配置 `idle-timeout`、`max-lifetime` 避免长连接堆积

## R-008 🟡 PageHelper / MyBatis-Plus 分页方言

**风险**：分页插件无原生瀚高方言，`mysql` 方言生成的 SQL 不兼容；`postgresql` 方言与瀚高 MySQL 兼容模式在细节上可能差异。

**影响**：分页查询结果错误、count SQL 报错、复杂分页 SQL 失效。

**缓解**：
- 默认配 `postgresql` 方言
- Pilot 中专项测分页（带 GROUP BY / ORDER BY / DISTINCT）
- 复杂分页场景重点回归
- 差异记录到 `fix-issue/`

## R-009 🟡 JSON 函数迁移量

**风险**：MySQL `JSON_EXTRACT` / `->` / `->>` 的路径语法（`$.a.b[0]`）与瀚高（PG 系）的 `#>` / `jsonb_path_query` 路径语法（`{a,b,0}` 或 jsonpath）完全不同，不是简单替换。

**影响**：JSON 查询迁移量大，返回值类型差异（text vs jsonb），业务逻辑受影响。

**缓解**：
- Stage 0 识别所有 JSON 查询位置并清点工作量
- 按位置逐一改写并写专项集成测试
- 字段类型统一用 `jsonb`
- 差异记录到 `fix-issue/`

## R-010 🟢 执行计划差异

**风险**：相同 SQL 在瀚高下执行计划可能显著不同，部分查询变慢。

**影响**：性能波动，非功能问题。

**缓解**：
- **不在本方案范围内调优**
- 记录明显慢查询到单独清单
- 若影响可用性，上升为阻塞项单独立项处理

## R-011 🔴 存储过程 / 触发器无法兼容

**风险**：瀚高基于 PG，存储过程语言为 PL/pgSQL，与 MySQL 存储过程语法完全不同，**不是调整而是重写**。触发器语义也不同。

**影响**：业务逻辑失效，迁移成本极高。

**缓解**：
- Stage 0 盘点全部存储过程 / 触发器使用情况
- 默认策略：**上移到 Java 层**，不在数据库层重写
- 无法上移的部分上升为独立立项，不纳入本方案
- 确认无存储过程依赖后再进入 Stage 2

## R-012 🟡 字符集与 collation

**风险**：MySQL `utf8mb4_general_ci` 大小写/重音不敏感比较；瀚高默认 collation 大小写敏感，业务中隐式依赖"大小写不敏感匹配"的查询会出错。

**影响**：WHERE 匹配失败、UNIQUE 约束行为变化、ORDER BY 顺序变化。

**缓解**：
- Stage 0 识别大小写不敏感依赖点（登录账号、邮箱、关键字搜索）
- 显式加 `LOWER()` 比较，或使用 `citext` 扩展 / `COLLATE` 子句
- Pilot 中专项测
- 唯一索引统一按 `LOWER(col)` 建函数索引

## R-013 🟢 EXPLAIN 输出格式差异

**风险**：运维脚本或监控若解析 EXPLAIN 文本输出，瀚高格式与 MySQL 完全不同。

**影响**：周边工具兼容问题，非主路径。

**缓解**：
- 记录受影响工具清单
- 非本方案范围，单独立项处理

## R-014 🟡 外部模板 / 非 Mapper SQL 扫描遗漏

**风险**：工程内可能存在配置文件里的 SQL 模板、`resources/sql/*.sql` 脚本、`@Query` 注解 SQL、代码字符串拼接 SQL，扫描时遗漏。

**影响**：运行时报 SQL 语法错误。

**缓解**：
- Stage 0 SQL 仓清点必须覆盖：Mapper XML / 注解 SQL / 脚本 sql / Flyway 迁移 / 字符串拼接
- 全仓 `grep` 常见 MySQL 专有关键字验证（`DATE_FORMAT`、`IFNULL`、`LIMIT ?,?` 等）
- 扫描结果列入 SQL 清单附录

## R-015 🟡 脚本重载类型缺口

**风险**：瀚高兼容脚本对 `IF` / `IFNULL` 等函数的类型重载可能存在缺口（如 `IFNULL(int, bigint)` / `IFNULL(numeric, text)` 未覆盖），调用时 "function does not exist"。

**影响**：运行时报错、调用失败。

**缓解**：
- Pilot 专项测 `IF` / `IFNULL` / `COALESCE` 在各类型组合下的行为
- 缺口优先改写为原生 `COALESCE` + 显式 CAST
- 不修改厂家脚本，而是在应用侧改写
- 缺口清单记录到 `fix-issue/`

## R-016 🟡 AUTO_INCREMENT 迁移方式选择

**风险**：MySQL `AUTO_INCREMENT` 可迁移为瀚高的 `IDENTITY`（SQL 标准、PG 10+）或 `SERIAL`（隐式 sequence），两者语义略有差异（OWNED BY、权限、重置方式）。选错可能导致序列残留或主键冲突。

**影响**：主键冲突、序列权限问题、导出导入时重复。

**缓解**：
- Stage 3 统一定型：**推荐 `GENERATED BY DEFAULT AS IDENTITY`**（SQL 标准，干净）
- 如历史遗留用 `SERIAL`，保持一致
- 迁移时显式 `ALTER SEQUENCE ... RESTART WITH` 调整起始值
- 决策记录到 `docs/references/mysql-to-highgo-type-mapping.md`

## R-017 🟡 兼容脚本版本管理漂移（per A4）

**风险**：厂家兼容脚本在多个工程间共享，但各工程注入时间 / 版本不一致，可能出现"A 工程验证通过的函数在 B 工程缺失"，问题难以定位。

**影响**：同一代码在不同工程行为不一致；Pilot 结论不可复用；修复追溯困难。

**缓解**：
- 在兼容脚本中注入 `mysql_compat_version()` 函数返回版本号 / commit hash
- Pilot 注入兼容脚本后执行 `SELECT mysql_compat_version()` 记录 baseline 到迁移报告
- 脚本任意改动必须 bump 版本号，同步更新注入物
- Stage 5 验收清单追加"兼容脚本版本标记一致"检查项
- 版本漂移记录到 `fix-issue/`

## R-018 🟡 Flyway 防护语法在瀚高下的兼容性（per C6）

**风险**：Flyway 迁移脚本常用的防护语法在瀚高（PG 系）支持度不一致：
- `CREATE TABLE / INDEX / SCHEMA IF NOT EXISTS`：PG 9.1+ 支持
- `ALTER TABLE ADD COLUMN IF NOT EXISTS`：PG 9.6+ 支持
- `ALTER TABLE ADD CONSTRAINT IF NOT EXISTS`：**PG 不支持**（约束无 IF NOT EXISTS 变体）
- `DROP ... IF EXISTS`：普遍支持

**影响**：历史 MySQL Flyway 脚本直接复用会在 ADD CONSTRAINT 处失败；重复执行不幂等。

**缓解**：
- Stage 3 §3.8 增加"防护语法冒烟测试"环节，建表 / 加列 / 加约束各跑一遍
- 在 `docs/references/mysql-to-highgo-syntax-mapping.md` DDL 节，对不支持的变体标注 🔄（需改写）/ ❌（不支持）
- `ADD CONSTRAINT` 替代方案：`DO $$ BEGIN IF NOT EXISTS (SELECT FROM pg_constraint WHERE conname='xxx') THEN ALTER TABLE ... ADD CONSTRAINT xxx ...; END IF; END $$;`
- 实测失败的具体场景记录到 `fix-issue/`
- 新迁移脚本模板预置瀚高可用的防护写法
