# v1.0.0 路线图与 Pilot 出口条件

## 1. 当前状态

- **版本**：v0.2.0（2026-04-21 目标库切换版）
- **成熟度**：文档 SOP 完成 / Skill 骨架完成 / **未经任何真实工程验证**
- **下一步目标**：通过 Pilot 工程 `stream_keywords_search` 回灌真实案例，升级到 v1.0.0
- **重要事件**：2026-04-21 目标库从 GaussDB B 兼容模式切换到瀚高 v4.1.5（决策见 `../decisions/2026-04-21-target-db-highgo-v4.md`），全工具包整改至 v0.2.0

## 2. 阶段划分

### 2.1 Pilot 执行期（v0.2.x）

**入口**：`stream_keywords_search` 工程启动改造分支 `feature/db-migration-highgo`
**执行方**：在 `stream_keywords_search` 工程内按五段式 SOP 推进，**本工具包作为依赖被软链消费**
**回灌动作**：
- 每个 Stage 完成后，把遇到的盲点、未覆盖场景、SOP 表述不清处记回本工具包
- 通用踩坑 → 根目录 `fix-issue/`（产品侧）
- 工具包自身流程问题（Skill 触发不准、模板字段缺失）→ `project-docs/fix-issue/`
- 新发现的方言差异点 → `docs/references/*.md`
- 新发现的风险 → `docs/risks/known-risks-highgo.md`
- 兼容脚本缺口或 Bug → `docs/references/highgo-v4.1.5-mysql-compat-functions.md` 缺口列 + R-017 版本管理流程
- 每次回灌走 PR，`CHANGELOG.md` 记 v0.2.x 增量

### 2.2 框架收敛期（v0.3.0）

**触发条件**：Pilot 第一次全流程跑通（Stage 0 → Stage 5 全绿）
**动作**：
- 基于 Pilot 真实产出，精化所有 Skill 的"执行步骤"从大纲到可执行指令
- 补齐 SOP 反馈表模板（当前 `master-plan.md §8.2` 有引用但未建立）
- 评估 `scripts/tools/` 是否需要（如扫描脚本、DDL 对照脚本）
- 校准兼容脚本版本与缺口清单（R-017）

### 2.3 v1.0.0 正式版

**出口条件**（全部满足才发布）：
1. Pilot 工程通过五段式 SOP 验收，产出完整验收报告
2. 至少 **1 个**非 Pilot 工程用 v0.3.x 独立跑通（无需 Pilot 维护者介入）
3. `fix-issue/` 至少沉淀 5 条通用踩坑条目（证明跨工程复用性）
4. `references/` 四份对照表被至少一次真实改造引用并验证正确
5. `references/highgo-v4.1.5-mysql-compat-functions.md` 的全部已知缺口（IF/IFNULL 类型重载、DATE_FORMAT 递归、TO_DAYS 偏差）被至少一次真实改造验证
6. `CHANGELOG.md` 有完整迭代记录
7. 所有 Skill 的 `description` 在实际 Claude Code 会话中验证触发准确

## 3. 非目标（本路线图不做）

- ❌ 扩展到 Oracle / DB2 等其他目标库（当前只做 MySQL → 瀚高 v4.1.5）
- ❌ 自动化代码改写工具（`dialect-rewrite` 只出建议，不做 AST 级 codemod）
- ❌ 数据迁移相关能力（见 `master-plan.md §10`）

## 4. 风险

- Pilot 周期可能超出 2 周——若超期，评估是否先发 v0.2.0-beta 让更多工程试用
- 软链分发方式在 Windows 上不可用——当前消费方全为 macOS/Linux，若未来变化需重新评估（见 `decisions/2026-04-21-skill-symlink-distribution.md`）
