# 瀚高国产化改造标准业务流程

## 一句话概括

把一个使用 MySQL 的应用工程，按"调研 → 测试兜底 → 环境切换 → 结构对齐 → 语义对齐 → 验收交付"六个阶段推进到以瀚高 v4.1.5 为运行库，过程中不做数据迁移、不改架构，只改应用侧 SQL 方言。

## 参与角色

| 角色 | 业务职责 |
|------|---------|
| 工具包维护者 | 方法论定义与版本演进，收集踩坑并回灌到方法论与对照库 |
| Pilot 工程 | 首批按方法论完成全流程的工程，验证工具包可用性并产出反馈 |
| 后续消费方工程 | 接入已被 Pilot 验证过的方法论，独立推进自身改造 |
| 工程改造负责人 | 单个工程内主导全流程、维护工程内事实与决策记录、签字验收 |
| 数据库管理员 | 负责目标库实例交付、兼容能力注入、权限授予 |
| 业务验收方 | 在 MySQL 基线和瀚高终态分别确认功能一致，签字出具验收结论 |

**关键约束**：工具包维护者与工程改造负责人是两个不同角色，前者面向"所有工程通用问题"，后者面向"单工程落地问题"。两者通过"踩坑回灌"机制连接。

## 流程步骤

### 第一阶段：立项与基线

**触发条件**：工程纳入国产化改造范围，已拉出独立改造分支。

**关键动作**：
1. 确认目标库版本与部署形态
2. 盘点工程持久层依赖的广度与深度
3. 扫描源码中对源库特性（函数、语法、类型、保留字）的依赖
4. 评估现有测试的覆盖缺口
5. 清点 SQL 仓总量（静态 SQL、动态片段、原生查询、硬编码）

**产出**：三件套事实文档——基线现状、风险矩阵、测试缺口。三件套只陈述事实，不下判断、不写方案。

**出口条件**：三件套全部就绪，风险矩阵中每条风险都有初步分级（阻塞 / 需改 / 可忽略）。

### 第二阶段：测试兜底

**触发条件**：第一阶段测试缺口文档标出"零覆盖关键路径"。

**关键动作**：
1. 对零覆盖的关键持久层方法补写集成测试
2. 测试必须连真实源库跑，不接受任何替代方案
3. 在源库环境下确认全部测试通过，形成**基线快照**

**产出**：补齐的测试用例集合 + MySQL 基线测试结果记录（用例数、通过数、耗时）。

**出口条件**：关键路径全部有集成测试覆盖且在源库下全绿。

**业务意义**：这一阶段是后续阶段的"防回退护栏"——任何改造造成功能破坏都会在这里被捕获。

### 第三阶段：依赖与配置切换

**触发条件**：源库下测试基线已绿。

**关键动作**：
1. 切换数据库驱动依赖坐标、连接串协议、连接池方言声明
2. 目标库中注入**厂家兼容能力**（让部分源库函数在目标库中保持原样可用）
3. 执行一组冒烟验证，包括正向（兼容能力确实生效）和反向（未覆盖的缺口确实报错）两类
4. 记录兼容能力的版本号，与工具包声明的版本号三方对齐

**产出**：可连通目标库、已注入兼容能力、冒烟通过的配置。

**出口条件**：工程能以目标库启动；正向冒烟 SQL 全通过；反向冒烟 SQL 按预期报错；兼容能力版本号一致。

**业务意义**：此阶段区分"免改部分"与"必改部分"——凡是兼容能力覆盖到的源库函数，后续方言适配阶段可以不动；未覆盖的才进入人工改写。

### 第四阶段：Schema 迁移

**触发条件**：目标库环境已就绪。

**关键动作**：
1. 基于源库结构生成目标库对照稿（人工审阅）
2. 在目标库中独立创建结构迁移路径（不改动源库历史迁移记录）
3. 针对目标库对结构保护语法的支持差异做一组冒烟
4. 核对表数、索引数等结构完整性

**产出**：目标库中完整就位的表、索引、约束、初始数据。

**出口条件**：结构与源库对齐；保护语法冒烟通过或已文档化例外；可重复执行无副作用。

### 第五阶段：方言适配

**触发条件**：Schema 已就位，且第三阶段已明确"未被兼容能力覆盖的缺口"。

**关键动作**：按风险矩阵分类别改写 SQL，每类改完后立即在目标库下跑对应测试：

- 标识符引用差异
- 分页语法差异
- 插入冲突处理差异
- 批量更新/删除差异
- 多表更新语义差异
- 查询提示与优化器介入差异
- 数据类型与函数语义的剩余差异
- 服务端存储过程与触发器的处理策略

每类改造独立成 commit，附测试绿证据。

**产出**：全部方言差异已在应用层或 DB 层抹平，目标库下用例集可完整运行。

**出口条件**：风险矩阵所有"需改"项状态为已关闭或已延迟决策（decision-deferred）。

**业务意义**：这是工作量最集中的阶段。节奏控制原则——**先分类、后批量**，避免在散点 SQL 上反复跳跃。

### 第六阶段：回归与交付

**触发条件**：方言适配完成。

**关键动作**：
1. 在源库下重跑一次测试（确认基线未被破坏）
2. 在目标库下跑全量回归（确认新环境可用）
3. 对两轮结果做用例数、通过数、耗时对比
4. 启动冒烟确认运行时无异常
5. 检查目标库中结构完整性
6. 产出验收报告骨架（目标库版本、各阶段耗时、改造范围、风险关闭情况、测试对比、配置变更、未解决问题、回滚方案）
7. 工程侧踩坑回灌——将通用性问题推送到工具包踩坑库

**产出**：验收报告（人工补全结论与签字）、PR、踩坑回灌材料。

**出口条件**：双环境测试全绿 + 启动冒烟通过 + 验收报告签字完成。

## 状态流转

```
工程入池
   ↓
第一阶段完成（三件套就位）
   ↓
第二阶段完成（源库基线绿）← 回退触发器：任何后续阶段破坏基线，回到这里确认
   ↓
第三阶段完成（目标库连通 + 兼容能力注入）
   ↓
第四阶段完成（结构对齐）
   ↓
第五阶段完成（方言适配完毕）← 回退触发器：风险矩阵有未关闭阻塞项，回到这里改写
   ↓
第六阶段完成（双环境全绿 + 签字）
   ↓
改造交付
```

**关键约束**：前一阶段未通过出口检查不得进入下一阶段。出口检查清单是"阶段闸门"，不是"形式勾选"。

## 异常路径

### 测试基线不绿（第二阶段）
业务处理：改造暂停，先修复测试或补齐缺失用例。基线不绿意味着后续改造的对错都无法判断，强推无意义。

### 风险矩阵存在"无解"方言差异（第五阶段）
业务处理：打上"延迟决策"标记，由工具包维护者介入评估是否为共性问题。共性问题 → 回灌工具包方法论；工程特有 → 单独制定规避方案。阻塞级别高的问题可导致该工程暂缓交付。

### 兼容能力版本不一致（第三阶段或第六阶段）
业务处理：以工具包声明的版本为基准，目标库重新注入。版本漂移会引起"同一段 SQL 在不同环境表现不同"，必须在交付前闭环。

### 目标库下功能与源库不一致（第六阶段）
业务处理：先区分是"改造过程中引入的回退"还是"已声明过的目标库特有行为"。前者必须回到第五阶段修正；后者需在验收报告中显式标注且经业务验收方签字认可。

### Pilot 烟测未通过（Pilot 阶段特有）
业务处理：工具包本身未达到可交付状态，**所有后续消费方工程的接入暂缓**，由工具包维护者先修复交付物。Pilot 烟测是"工具包可用性"的守门员，不是工程迁移本身的一部分。

## 完整示例

某工程从入池到交付的完整走位（以 Pilot 工程视角）：

1. **Day 1 上午**：工程改造负责人拉改造分支，执行第一阶段动作——确认目标库 v4.1.5 已由数据库管理员部署就绪且已注入厂家兼容能力；盘点发现工程持久层混用两种 SQL 承载形式（配置式映射与注解式声明），均依赖反引号标识符与 `LIMIT m,n` 分页；扫描产出风险矩阵 87 条。

2. **Day 1 下午**：执行测试缺口评估，发现 12 个核心查询方法零集成测试覆盖。

3. **Day 2 全天 + Day 3 上午**：补 12 个集成测试用例，在源库下跑全绿，记录基线快照（用例数 340、通过 340、耗时 186 秒）。

4. **Day 3 下午**：切换驱动与连接串，连通目标库；数据库管理员配合执行兼容能力注入；跑 7 条冒烟——4 条正向（含日期格式化函数）全通过，3 条反向（含未覆盖的函数重载）按预期报错，兼容能力版本号返回与工具包声明一致。

5. **Day 4**：Schema 对照稿人工审阅后应用，目标库结构就位；结构保护语法冒烟发现 1 处目标库不支持的变体，文档化例外。

6. **Day 5 ~ Day 7**：方言适配按类别推进——反引号替换、分页改写、冲突处理改写、批量操作调整，每类独立 commit 并跑对应测试。风险矩阵 87 条逐条关闭，其中 3 条标记延迟决策（与业务验收方达成共识）。

7. **Day 8 上午**：源库基线重跑确认未被破坏；目标库全量回归 340 条用例全绿（耗时 203 秒，未超警戒线）；启动冒烟无异常。

8. **Day 8 下午**：产出验收报告骨架，业务验收方签字；PR 提交；踩坑回灌——发现 2 条通用问题（某函数重载缺失、某保护语法目标库不支持），推送到工具包踩坑库供后续工程规避。

总工期约 8 个工作日。Pilot 工程产出的踩坑回灌使后续消费方可以跳过这两个坑。
