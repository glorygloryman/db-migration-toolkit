# db-migration-stage4-plan-rewrite

## 触发场景

- `/db-migration-dialect-rewrite` 已生成 `YYYY-MM-DD-stage4-*-rewrite.md` 文档
- 需要将文档中的改写条目拆解为可逐条执行的原子任务
- 用户希望按任务粒度在新会话中逐步执行

## 前置条件

- 项目目录下存在 `project-docs/plans/YYYY-MM-DD-stage4-*-rewrite.md`

## 执行步骤

### 1. 定位 stage4 文档

在 `<project>/project-docs/plans/` 下找到最新的 stage4 改写清单文档。若有多个，列出供用户选择。

### 2. 解析文档结构

读取文档，提取以下信息：
- 所有改写条目（按 C-01、C-02、D-01 等编号）
- 每个条目的：文件路径、行号、改写内容摘要、依赖关系
- 文档中的"执行顺序"部分
- 文档中的"验证清单"部分

### 3. 生成原子任务清单

将文档内容拆解为原子任务，粒度原则：
- **每个原子任务 = 一次独立会话可完成的工作单元**
- 按文件+方法维度拆分（如 C-01-1 = RegionMapper.xml 的反引号替换）
- 标注任务间依赖关系（如 C-02-1 依赖 D-01 已完成）
- 合并可同时执行的条目（如 C-01 + D-02 涉及同一文件的同一类改动）

每个原子任务包含：
```
- [ ] <任务编号>: <任务标题>
    文件: <涉及的文件路径>
    改动摘要: <一句话描述>
    依赖: <前置任务编号，无则填"无">
    风险: <低/中/高>
    对应集成测试: <需验证的测试方法名>
```

### 4. 生成标准化执行提示词

为每个原子任务生成一段标准化的执行提示词模板，用户可直接复制到新会话使用：

```
读取 @<plan文档路径>，执行 <任务编号> 的改写。
如有需要方案抉择的情况，请向我提出之后再施行。
完成后，运行以下对应的集成测试验证：<测试方法列表>
将测试不通过的集成测试用例告知我，你不用去管报错原因，只需要告知我有哪些方法报错就可以。
```

### 5. 产出执行计划文件

写入 `<project>/project-docs/plans/YYYY-MM-DD-stage4-execution-plan.md`：

```markdown
# Stage 4 执行计划

> 源文档：<stage4-rewrite文档链接>
> 生成时间：<当前日期>
> 总任务数：N
> 已完成：0 / N

## 任务列表

### 批次 1（无依赖，可立即执行）
- [ ] T-001: <标题> ...
- [ ] T-002: <标题> ...

### 批次 2（依赖批次 1）
- [ ] T-003: <标题> ...

...

## 执行提示词模板

### T-001: <标题>
```
<标准化提示词>
```

### T-002: <标题>
```
<标准化提示词>
```

...

## 进度追踪

| 任务 | 状态 | 完成日期 | 备注 |
|------|------|----------|------|
| T-001 | 待执行 | - | - |
| T-002 | 待执行 | - | - |
```

### 6. 控制台摘要

输出：
- 总任务数及分批情况
- 风险等级分布
- 提示用户："执行计划已生成。请使用 `/db-migration-stage4-execute-task` ，并指定任务编号，逐条执行，或手动复制执行提示词到新会话。"

## 约束

- 不直接改代码，只生成执行计划
- 任务编号使用 `T-001` 格式，与原文档的 C-01 等编号建立映射关系
- 每个任务必须标注对应的集成测试方法
- 保留原文档中"需决策"条目的决策提示
- 如果某条目原文档标注为"不改"或"纯验证"，仍生成任务（验证任务），不可跳过
