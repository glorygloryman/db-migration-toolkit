# db-migration-schema-convert

## 触发场景

- Stage 3 开始，已有 MySQL Schema 导出
- 需要生成 瀚高 Flyway 脚本初稿

## 前置条件

- 已有 MySQL Schema 导出文件（`mysqldump --no-data` 或手写 DDL）
- 已完成 Stage 0（baseline + risk-matrix）
- 已读 `docs/references/mysql-to-highgo-type-mapping.md`

## 输入

- MySQL Schema SQL 文件路径
- 目标输出路径（`<project>/src/main/resources/db/migration/highgo/`）
- 本工程的类型映射决策（若已有）

## 执行步骤

### 1. 解析 MySQL DDL

- 按 `CREATE TABLE ... ;` 分段
- 提取每表：列定义、索引、约束、表选项

### 2. 表级改写

- 去除 `ENGINE=InnoDB`、`DEFAULT CHARSET=xxx`、`COLLATE=xxx`
- `AUTO_INCREMENT=100` 起始值：改为 PG `GENERATED BY DEFAULT AS IDENTITY (START WITH 100)` 或独立 `SEQUENCE`
- 表注释 `COMMENT='...'`：PG 语法不支持 inline，改为独立 `COMMENT ON TABLE t IS '...';`
- 添加 `IF NOT EXISTS`（CLAUDE.md §3.6）

### 3. 列级改写

按 `references/mysql-to-highgo-type-mapping.md` 映射（PG 语法）：
- `MEDIUMINT` / `INT UNSIGNED` → `INTEGER`（或 `BIGINT` 升格）
- `TINYINT(1)` → `BOOLEAN`（或按业务保留 `SMALLINT`）
- `TINYTEXT` → `VARCHAR(255)`
- `MEDIUMTEXT` / `LONGTEXT` → `TEXT`
- `DATETIME` → `TIMESTAMP`（或 `TIMESTAMPTZ`，按时区决策）
- `BLOB` / `MEDIUMBLOB` / `LONGBLOB` → `BYTEA`
- `JSON` → `JSONB`（PG 推荐，查询性能更好）
- `ENUM` → `VARCHAR(n) + CHECK` 约束（附带注释说明原枚举值）
- `SET` → 标记待人工决策（拆关联表 或 VARCHAR）
- `AUTO_INCREMENT` 列 → `GENERATED BY DEFAULT AS IDENTITY`
- `UNSIGNED` → 升格（PG 无 UNSIGNED 概念）
- 其他按表保留

列注释 `COMMENT '...'`：改为独立 `COMMENT ON COLUMN t.c IS '...';` 语句。

### 4. 保留字处理

检测列名是否命中 瀚高/PG 保留字清单：
- 命中 → 加双引号（`"user"`）或建议改名（加注释 `-- TODO: 原列名冲突保留字，建议改名为 xxx`）
- 默认策略：加双引号（改名影响代码面，决策风险大）

### 5. 索引处理

- 普通 `KEY idx (col)` → `CREATE INDEX IF NOT EXISTS idx_x ON t(col)`
- `UNIQUE KEY` → `UNIQUE` 约束或 `CREATE UNIQUE INDEX`
- **前缀索引 `KEY idx (col(10))`**：标记 TODO，建议改为函数索引或去前缀
- `FULLTEXT INDEX`：标记 TODO，单独评估
- 索引名保持一致

### 6. 外键与约束

- 保留 `FOREIGN KEY ... ON DELETE / ON UPDATE`
- `CHECK` 约束保留

### 7. 大小写规范化

- 所有表名、列名、索引名统一小写
- 若原 Schema 有大写字母，输出中统一转小写并加警告注释

### 8. 分文件写入

- 所有表 DDL → `V{timestamp}__highgo_init_schema.sql`
- 如果表很多（> 50），拆成多个版本号递增的脚本（按模块聚合）
- 索引单独 → `V{timestamp+1}__highgo_init_indexes.sql`（可选）
- **冒烟验证**：生成的 DDL 必须经过 Task 9 §3.8 的防护语法冒烟（`IF NOT EXISTS` / `IF EXISTS` 在瀚高下的实际行为验证），未验证不得进入 Stage 4

### 9. 输出转换报告

另写 `project-docs/reports/YYYY-MM-DD-schema-convert-report.md`：
- 每表转换摘要
- 所有 TODO 项清单（保留字、前缀索引、ENUM、SET、FULLTEXT）
- 类型改写统计
- 需人工决策项

### 10. 回写风险矩阵建议

定位最新的 `<project>/project-docs/facts/*risk-matrix*.md`，筛出由 Stage 3 承接的 DDL / Schema / seed / 类型映射 / 索引 / 注释 / 自增相关风险，按以下三类给出回写建议：

1. `stage3-closeable`：风险边界仅限 DDL / Schema 资产，且 schema convert report、目标 DDL、DDL asset test、schema diff 或 DDL guard smoke 已能证明风险关闭。
2. `stage3-partial`：Stage 3 只完成 DDL 部分，仍需 Stage 4 / Stage 5 通过 Mapper / DAO 真库读写或业务回归验证。
3. `stage3-deferred`：本轮不迁移、需决策或证据不足，必须保留为 `pending` / `blocked` / `decision-deferred` 中的合适状态。

可建议 Stage 3 直接关闭的典型风险：

- `AUTO_INCREMENT` → identity / sequence，含历史起始值策略。
- `ENGINE` / `DEFAULT CHARSET` / `COLLATE` / `ROW_FORMAT` 移除。
- 内联 `COMMENT` / 表级 `COMMENT='...'` → `COMMENT ON TABLE/COLUMN`。
- `USING BTREE` / 内联 `KEY` → 目标库索引语法。

不得仅凭 Stage 3 关闭的典型风险：

- `tinyint` / `jdbcType=TINYINT` 等 Mapper 参数绑定风险。
- `datetime` / `timestamp` / `jdbcType=TIMESTAMP` 等时间读写语义风险。
- `longtext` / `jdbcType=LONGVARCHAR` 等大字段读写风险。
- 字符集 / collation 差异引出的大小写、排序、`LIKE` 查询语义风险。

对 `stage3-closeable` 条目，先输出回写计划，列明：

- 风险 ID
- 建议状态
- 关闭依据
- 需要写入备注的验证命令 / 报告路径 / 目标 DDL 证据

只有在用户确认后，才回写 `risk-matrix.md` 的状态和备注；不得默认批量关闭风险。

## 输出

- `<project>/src/main/resources/db/migration/highgo/V*.sql`
- `<project>/project-docs/reports/YYYY-MM-DD-schema-convert-report.md`
- `<project>/project-docs/facts/*risk-matrix*.md`（仅在用户确认 Stage 3 回写计划后更新）

## 约束

- **不自动修改 `db/migration/mysql/` 任何脚本**
- 输出脚本必须含 `IF NOT EXISTS` 防护
- 无 `DROP` / `TRUNCATE`
- TODO 项必须显式注释，不暗改
- 不得在未执行验证或未取得用户确认时批量关闭风险矩阵条目

## 人工复核要点

- 每个 TODO 项必须给出决策（改写 / 保留 / 推迟）
- 类型映射决策记录到 `project-docs/decisions/YYYY-MM-DD-type-mapping-strategy.md`
- 在 瀚高 v4.1.5 测试环境执行一次 Flyway migrate 验证
- `stage3-closeable` / `stage3-partial` / `stage3-deferred` 分类是否正确
- 风险矩阵关闭备注是否包含可追溯证据，而不是只写"Stage3 已处理"
