# db-migration-stage4-execute-task

## 触发场景

- 执行计划已由 `/db-migration-stage4-plan-rewrite` 生成
- 用户要执行执行计划中的某个具体任务

## 输入

- 任务编号（如 `T-001`），或关键词（如 `C-01`、`RegionMapper`）
- 若未指定，列出待执行任务供用户选择

## 前置条件

- 存在 `<project>/project-docs/plans/YYYY-MM-DD-stage4-execution-plan.md`

## 执行步骤

### 1. 读取执行计划

定位并读取最新的 `stage4-execution-plan.md`，以及对应的 stage4 改写清单源文档。

### 2. 解析目标任务

- 根据用户指定的编号或关键词定位任务
- 若任务已完成（已勾选），提示用户并确认是否重新执行
- 检查前置依赖任务是否已完成，未完成则警告用户

### 3. 读取源文档详情

从 stage4 改写清单源文档中读取该任务的完整改写说明，包括：
- 原 SQL 片段
- 建议改写的 diff
- 风险说明
- 参考文档链接

### 4. 执行改写

- 定位目标文件和行号
- 如果改写方案明确且唯一，直接执行
- 如果源文档标注"需决策"或存在多个方案，**必须向用户提出方案选择后再施行**
- 改写完成后展示 diff 摘要

### 5. 运行集成测试

- 仅运行该任务对应的集成测试方法（不是整个测试文件）
- 通过用户项目的测试命令执行（如 `mvn test -pl <module> -Dtest=<TestClass>#<method>`）
- 收集测试结果

### 6. 报告结果

向用户报告：
- 改写完成的文件和改动摘要
- 集成测试结果：通过/不通过
- 不通过的测试方法列表（只列出方法名，不分析原因）

### 7. 回写进度

更新 `stage4-execution-plan.md`：
- 勾选已完成的任务 `- [x]`
- 更新进度追踪表的状态和完成日期
- 更新顶部"已完成：X / N"计数

若测试不通过，不勾选任务，但在进度表中标注"测试未通过"。

## 输出

- 改写后的源文件
- 更新后的执行计划文件
- 控制台结果摘要

## 约束

- 改写必须严格遵循源文档的建议方案
- 不自行发明改写方案，不确定时向用户提问
- 集成测试只运行对应的具体方法，不跑全量
- 测试不通过时只报告方法名，不尝试修复
- 每次只执行一个任务
